# Design — add-cloud-sync

## Context

Binding prior decisions: `.research` bundles with the three-layer split the format doc defines for sync (source = immutable content-addressed; derived = regenerable, pipeline-versioned; user = append-only JSONL journals anchored to UUIDs + a handful of small LWW documents). Library-level stores follow the same shapes (`learning_state/` journals, `concepts.jsonl`, `reviews/<id>/document.md`). v1–v3 shipped zero server-side anything; keys live in the OS keychain; egress transparency is an established rule. research-format.md left one open question — "CRDT library choice (Automerge/Yjs/Loro), decided in sync design" — this document decides it. v4's collaboration section is gated on this change.

## Goals / Non-Goals

**Goals:** a library that exists identically on N devices; user data merges without destructive conflicts; everything leaving the machine E2E-encrypted; no accounts, no first-party server; resumable, offline-first, background; the substrate v4 shared workspaces sit on.

**Non-Goals:** realtime collaborative editing (presence/live cursors — v4+, needs a live channel this design deliberately omits); the web app; selective partial-bundle sync (whole bundles sync; only heavy caches are excluded); sync of the repo clone cache or container images (re-derivable by definition); any recovery backdoor for a lost passphrase.

## Decisions

1. **No CRDT library — journal union-merge.** This resolves the format doc's open question. Nearly all user data is already append-only JSONL where entries are immutable and self-contained (notes/chats/ink events, mastery/preferences/episodes, consents, experiment runs, graph/code-map overrides, v4 cards/threads). Two devices' journals merge by set union: dedupe on entry content-hash, order by (`at` timestamp, content-hash) for determinism. Every reader in the codebase already folds journals with edit/delete *events* rather than in-place mutation, so union-merge is semantically correct by construction. Rationale: Automerge/Yjs/Loro solve concurrent *rich-text* editing — a problem this data model was explicitly designed not to have; adopting one would add a heavyweight dependency to solve nothing.
2. **The exceptions are LWW + conflict copy.** Non-journal user files (`reading_state.json`, review `document.md`, workspace metadata): last-writer-wins by wall-clock with the losing version preserved as `<name>.conflict-<device>-<date>` and surfaced in the UI. Rationale: these files are either trivially re-doable (reading position) or too important to auto-merge (an edited review) — keeping both honest copies beats a wrong merge.
3. **Layer rules from the format table, enforced by a manifest.** Each bundle syncs via a manifest listing (path, layer, content-hash, size): **source** — content-addressed, encrypted, uploaded once, verified by hash; **derived** — small JSONs sync (cheap, saves re-ingestion on the second device), heavy caches excluded by rule (`embeddings.bin`, `graph.db`, `repos/`, images under `figures/` sync since they're small and save PDFium work); **user** — always syncs, always encrypted, journals via decision 1. A stale derived file on the receiving device is just the normal stale-stage case — the pipeline re-runs on open exactly as it does after an app upgrade.
4. **E2E encryption: XChaCha20-Poly1305, per-library key from Argon2id(passphrase).** Every object uploaded (user layer, source PDFs, manifests' user-file entries) is encrypted client-side; blob names on the remote are HMAC-derived from content hashes so the remote learns sizes and access patterns only. The passphrase never leaves the machine; the derived key lives in the OS keychain. Losing the passphrase means re-uploading from a device that still has the data — documented, no backdoor. Rationale: the learner-memory privacy designation ("E2E-encrypted-only") is a hard spec obligation; a dumb remote must be assumed hostile.
5. **Remotes are dumb blob stores behind one small trait — free by default, self-hostable by choice** (`get/put/list/delete` with atomic-put semantics via temp+rename or content-addressed keys + manifest swap). Two backends:
   - **S3-compatible** — the **primary recommended path is a Cloudflare R2 free-tier bucket** (10 GB free, no credit card for the free tier, five-minute setup, reachable from anywhere with zero infrastructure to run — E2E encryption means Cloudflare only ever stores ciphertext). The same backend fully supports **self-hosted MinIO** (one-click on Coolify or plain docker) as the equal-citizen option for users who prefer their own hardware and disk. AWS works too but is deliberately not the documented path — nothing here requires a paid cloud account.
   - **Local folder** — pointing it at iCloud Drive/Dropbox/Syncthing gives zero-config sync through infrastructure users already trust; also covers USB/NAS mounts.
   No WebDAV backend (cut: Nextcloud-class targets add protocol surface without adding a user story the two backends above don't already cover). No first-party server, no accounts; device pairing = same remote credentials + same passphrase. Rationale: R2-free-tier as the default gives every user a working, always-reachable remote at zero cost with nothing to operate; BYO-storage still means *bring your own* — including your own machine — and a shared bucket is already the v4 shared-library substrate.
6. **Sync is a resumable, interruptible reconcile loop** mirroring the staged-pipeline pattern: build local manifest → fetch remote manifest → diff → pull (merge journals, write LWW+conflicts) → push (upload blobs, then swap the manifest atomically). Per-library `sync_state/` (cache-class, never source of truth) records the last-reconciled manifest so interrupted syncs resume and unchanged files never re-upload. Sync runs in the background (app open, paper close, manual button) and *never* blocks reading — offline-first means the app is byte-identical in behavior with sync off.
7. **Deletions are tombstones, never remote deletes of user data.** Deleting a paper locally writes a tombstone to the library journal; other devices honor it (move to a local trash for a grace period). Blob garbage collection on the remote happens only for objects unreferenced by any manifest, from an explicit "clean remote" action. Rationale: sync bugs that destroy data are the one unrecoverable failure class; tombstones + explicit GC make destruction always deliberate.
8. **Concurrency control without a server: manifest compare-and-swap.** Manifests carry a monotonic generation + writing-device id; a push that finds the remote generation advanced re-pulls, re-merges, retries (bounded). Journal union makes the re-merge safe. Swap primitive per backend: S3 conditional writes (`If-None-Match` on generation-numbered manifest keys — supported by recent MinIO and R2) with list-and-verify as the portable fallback for older S3 servers; the folder backend uses lock-file + rename (best-effort, documented as single-writer-at-a-time).
9. **Trust boundaries in the UI**: configuring a remote shows exactly what leaves the machine (encrypted blobs, to which host), sync status is per-paper (synced/pending/conflict), conflict copies get a visible badge, and telemetry stays local (sync event kinds are content-free counts).

## Risks / Trade-offs

- [Silent data destruction] → decision 7 (tombstones, explicit GC), decision 2 (conflict copies), and a merge test suite: interleaved-device journal appends fold identically regardless of sync order (commutativity test).
- [Clock skew breaks LWW] → LWW only ever applies to low-stakes or conflict-copied files; journals (the valuable data) are order-insensitive by union. Timestamps are advisory, not load-bearing.
- [Folder-backend races (two devices via Dropbox)] → lock-file discipline + generation checks + journals-are-unions means the worst case is a redundant re-merge, not corruption; documented as eventually-consistent.
- [Self-hosted server reachability (MinIO behind a home network)] → sync is offline-first anyway: unreachable remote = "pending", never an error state; document Tailscale/VPN as the recommended way to reach a home Coolify/MinIO from outside; all failures retry with backoff.
- [Crypto is easy to get wrong] → boring, standard constructions only (XChaCha20-Poly1305 AEAD, Argon2id KDF, HMAC-SHA256 naming); no custom protocol; test vectors in the suite; passphrase-loss consequence documented in UI at setup.
- [Large first sync] → content-addressing dedupes PDFs; heavy caches excluded; progress + resumability from decision 6.
- [Remote quota/failures mid-push] → manifest swaps last, so a partial push is invisible to other devices; retry resumes from sync_state.
- [Scope creep toward a sync service] → the trait is three dumb verbs; anything needing a smarter server (presence, live editing) is explicitly v4+/web and out of scope here.

## Migration Plan

Purely additive: no format bump (bundles unchanged; `sync_state/` is local cache-class). Sync off by default; enabling it is per-library opt-in. Rollback = disable sync / previous app version; local data is always the complete source of truth. Remote layout is versioned (`layout_version` in the manifest root) for future evolution.

## Verification Plan (real infrastructure, no placeholders)

- Engine/merge/crypto: property tests + two-device simulation through a shared folder (the real protocol, two library dirs on one machine).
- S3 backend: developed against a local MinIO container (docker), then verified against **a real free-tier R2 bucket** (the primary documented user path — the owner's Cloudflare account has R2 access) and against the owner's **self-hosted Coolify MinIO** for the self-hosted path (over Tailscale when enabled; not needed until then).
- Folder backend: live smoke against an actual iCloud Drive directory.
