# Design — add-v3-hacker-workspace

## Context

Binding prior decisions: code under `app/` (Rust core `copilot-core` + Tauri 2 shell + React/shadcn/COSS/AI-Elements); `.research` bundles with immutable PDF, derived regenerable JSON, append-only UUID-anchored user data; versioned resumable pipeline stages; provider layer with tier routing, streaming + cancel, designed no-key states; v2 added the knowledge graph, learner memory, graph-first context assembly, and learning surfaces. `implementations/` and `experiments/` have been reserved bundle directories since v0 of the format. Performance budgets are CI-enforced release blockers (perf/budgets.toml). PDFium is process-global/single-threaded (irrelevant to v3 paths but binding on anything touching layout).

The v3 PRD's stated top risk is executing arbitrary code (generated or cloned). That concern shapes this whole design: one execution substrate, specced as its own capability, gating every run path.

## Goals / Non-Goals

**Goals:** runnable implementations for equations/algorithms in 5 languages; parameterized experiments with persisted results and AI discussion; end-to-end reproduction of papers-with-code on a curated corpus; code↔paper navigation; all execution containerized, consented, offline-by-default; everything stored in the open bundle format.

**Non-Goals:** cloud execution or GPU orchestration (local small-scale/verification runs only; larger-scale paths documented, not built); the web app (separate horizon, needs cloud sync); arbitrary-language support beyond the five named; autonomous multi-step agents editing repos (v4 territory); Windows container support beyond whatever Docker Desktop provides (document, don't engineer around).

## Decisions

1. **One sandboxed-execution capability, five consumers.** A single `sandbox` module in copilot-core owns container lifecycle: image selection, mount policy, resource caps (CPU/mem/time), teardown. Every run path (implementation kernel, experiment run, repro build/run) goes through it — there is deliberately no second way to execute anything. Rationale: the PRD risk is existential; one choke point is auditable, five ad-hoc runners are not.
2. **Container runtime via CLI subprocess (docker/podman autodetected), not a daemon SDK.** Orchestrate `docker run`/`podman run` with explicit flags (`--network=none` default, `--memory`, `--cpus`, `--pids-limit`, read-only rootfs where possible, tmpfs scratch, bind-mount only the specific bundle subdirectory read-write). Rationale: subprocess keeps us runtime-agnostic (podman = rootless bonus), avoids a heavyweight always-on dependency like bollard, and every invocation is loggable verbatim — observability is a spec requirement. Runtime absence is a designed state ("install Docker or Podman"), never a crash.
3. **Consent is per-scope and explicit, never blanket.** First run of each scope (this paper's implementations; this experiment; this cloned repo) shows exactly what will run, with what mounts, and that network is off; the grant is recorded as user data (append-only `consents.jsonl` in the bundle) and revocable. Network access is a separate, per-run, opt-in consent with the reason shown (e.g. "pip needs to download torch"). Rationale: PRD hard requirement; consent that people can't audit isn't consent.
4. **Implementations are user-data-class artifacts anchored to objects.** `implementations/<object-uuid>/<language>.{py,rs,…}` plus a sidecar `meta.json` (anchor hash, generation provenance, check status). Generated once via strong tier, then user-editable — edits never regenerate silently (same rule as graph overrides). Generated checks live next to the code (`checks.py` style, assert-based); passing them emits a mastery event (source "implementation") and flips the dashboard signal. Rationale: reuses the v2 anchoring + mastery pipeline instead of a parallel progress system.
5. **Experiments are event-sourced runs over one implementation.** `experiments/<experiment-uuid>/experiment.json` (name, anchor, parameter schema) + append-only `runs.jsonl` (params, stdout/metrics, duration, exit) + `discussion.jsonl` (reuses chat journal semantics keyed by experiment uuid). Graphs are derived client-side from runs (Recharts), never stored as images. AI discussion context = experiment definition + selected runs' metrics, assembled through the existing context.rs budget machinery with a new anchor kind. Rationale: append-only runs give crash-safety and side-by-side comparison for free.
6. **Reproduction is a resumable step pipeline, mirroring ingestion.** Steps: clone → detect env (uv > conda > docker heuristics) → build image → map code↔paper → run → verify → report. Each step versioned and persisted under `reproduction/` in the bundle (clone metadata, env spec, mapping, run logs, report.md), each observable (step events like `ingestion-progress`) and interruptible (kill container, keep partials). Report = template filled with verified/diverged metrics table + provenance. Rationale: users already understand the staged-pipeline mental model; resumability is what makes 30-minute repro sessions survivable.
7. **Code↔paper mapping is derived data with the same honesty rules as extraction.** Strong-tier pass over repo tree + paper objects produces `reproduction/code_map.json` (file/function/line-range ↔ object UUID, with confidence); low-confidence links styled as such; user corrections append-only (`code_map_overrides.jsonl`) and survive re-mapping. Repo browser reads the map; "where is Equation 12" = map lookup + editor-style file pane with line highlighting (CodeMirror 6, already-lightweight, no LSP).
8. **Curated corpus first.** Reproduction quality gates run against a small pinned list of well-behaved ML repos (the golden-corpus pattern); anything outside it still works but sets expectations ("unverified repo — steps may need manual help"). Rationale: PRD's environment-hell mitigation.
9. **Format bump 0.2.0 → 0.3.0 (additive).** Activates `implementations/`, `experiments/`, adds `reproduction/`. Unknown-file preservation covers older readers; same-major compatibility test extends the existing one.
10. **UI placement:** implementation tabs join the object panel (the v1 action strip that became deep-dive tabs); experiment workbench and repo browser are reader-shell panes like graph view/reading mode (escape-to-paper preserved); reproduction wizard is a route off the dashboard.

## Risks / Trade-offs

- [Arbitrary code execution] → decision 1–3: single substrate, container + no-network default + per-scope consent + resource caps; CI test asserts the no-consent path cannot reach `docker run` (unit-level guard on the choke point).
- [Environment hell] → curated corpus for gates (decision 8); uv-first detection (fastest, most deterministic); every failed step shows the exact command + log, interruptible, resumable.
- [Container runtime not installed] → designed empty state with install guidance; all non-execution features (code browsing, mapping, viewing cached implementations) still work.
- [GPU-dependence] → verification-scale runs only (seeded, small); report says "verification run" vs "full reproduction" explicitly — honest about what was checked.
- [LLM code quality] → generated implementations ship with generated checks and a "generated, unreviewed" banner until first successful check run; editing is expected, not exceptional.
- [Five languages × maintenance] → one canonical prompt template parameterized by language; Rust runs in the same container substrate (rustc image), no host toolchain assumptions.
- [Disk growth from clones/images] → clones live outside bundles in a library-level `repos/` cache keyed by remote+commit (bundles store references + the map, staying portable); image/layer cleanup surfaced in settings.

## Migration Plan

Additive only. 0.3.0 bundles open in v2 readers (same major; new dirs preserved untouched). No data migration. Rollback = previous app version; consents and experiments remain inert files.

## Open Questions

- ~~Default container images per language~~ **Resolved (task 1.1 spike, 2026-07-02, docker 27.4 via OrbStack):** per-language official images, pulled on first consented use — `python:3.12-slim` for Python/checks, `rust:1-slim` for Rust, framework images (`pytorch/pytorch`, `tensorflow/tensorflow`, JAX on CUDA-less `python:3.12-slim` + pip with network consent) for the DL variants; no custom kernel image to maintain. Verified flag set (all asserted from inside the container): `--network=none --memory --cpus --pids-limit --read-only --tmpfs /tmp --security-opt no-new-privileges --cap-drop=ALL`, bundle subdir mounted `:ro` with only the output subdir `:rw`. Findings: warm container start→output 0.35 s (budget-friendly); OOM kill surfaces as exit 137 (map to "limit-killed"); `docker kill` returns in <100 ms (kill-anytime holds); `--env-clear` does not exist and is unnecessary — docker containers never inherit host env (7 image-standard vars observed). Image *pulls* are a host-side network operation triggered by the app → pulls require the network consent flow even though the run itself stays offline.
- Whether experiment graphs need server-side (core) metric parsing beyond stdout-JSON convention — start with a documented `{"metric": value}` stdout convention, revisit if the corpus demands more.
- Apple `container` CLI (macOS 26) as a third runtime — detect if trivial, otherwise document Docker/Podman only.
